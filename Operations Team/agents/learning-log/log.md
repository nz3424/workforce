## 2026-07-17
- [Transformers/Backprop] Residual connections act as a gradient "floor" via the chain rule expanding into a sum over 2^L skip/pass-through paths — not a literal `+1` you can trace back — and that floor only protects gradient flow *between* blocks, never a block's own internal weights.
- [Transformers/Backprop] The residual `+1` guards against vanishing gradients but not exploding ones (since `1+f' > f'` for `f'>0`) — normalization, init, and clipping do the exploding-gradient job separately.
- [Transformers/FFN] The FFN block works as a per-token key-value lookup — `W1` rows are detectors, `W2` columns are the associated values — and confirmed this mapping against the real `llm-training/src` code (GELU, 4x `d_ff`, pre-norm structure).
- [Transformers/Attention] Softmax's Jacobian is symmetric (it's the Hessian of log-sum-exp), and a confident/peaked softmax has a near-dead local gradient — the actual mechanistic reason `1/√d_k` scaling matters for training, not just numerics.

## 2026-07-20
- [llm-training] At inference the weights are frozen and only the attention/MLP activations get recomputed per input (they're functions of the tokens) — but "recomputed" doesn't mean "from zero," since KV caching and prefix reuse skip re-deriving anything that hasn't changed.
- [llm-training] "Model memory" is three distinct mechanisms people conflate: in-context recall (same conversation — the info is just still sitting in the window), retrieved memory (injected as tokens across conversations), and training-baked knowledge — none of them is a live lookup into the weights.
- [quantm-learning] A token's K and V are per-layer (each block has its own k_proj/v_proj) and permanently fixed once computed given frozen weights (the causal invariant) — so KV caching only avoids re-deriving unchanged values, it never stores new information.
- [quantm-learning] You can't speed up generation by running only the latest token through the model — attention for the new token structurally needs every past position's K/V at every layer, so that information must exist; the cache keeps it, slicing the input to one token just destroys it.
- [quantm-learning] GQA shrinks the KV cache by the n_heads/n_kv_heads ratio (e.g. 32→8 = 4x) purely by having fewer K/V projections — Q keeps its full head count and dimensionality untouched.
- [quantm-learning] MoE's router scores all experts for every token (softmax over router logits — same math as the output head) before the discrete top-2 cut, so the router's weights get gradient from every token while only the two selected experts' internal weights do.
- [quantm-learning] Attention has to run before the FFN inside a block because attention is the only sub-block that can move information between positions — the FFN is applied per-position, so it needs cross-token context already folded in.
- [quantm-learning] PyTorch's `.grad` accumulates (`+=`) across `backward()` calls, so a missing `zero_grad()` sums gradients across steps into one tensor — it's not a clean one-step lag, it's a running sum.
- [quantm-learning] Byte-level BPE guarantees exactly a 256-symbol base vocabulary (it starts from raw UTF-8 bytes, not Unicode code points) — that fixed floor is what keeps the embedding/head the one full-precision component even in a ternary-weight model.
- [quantm-learning] In photonics, optical addition is passive and nearly free (beams just superpose) while optical multiplication needs an active nonlinear device — this add-cheap/multiply-expensive asymmetry is the core reason ternary weights fit photonic accelerators.
- [quantm-learning] Ternary weights don't remove thermal drift or crosstalk as physical effects — with only 3 states instead of 256 levels, a given amount of drift eats a far smaller fraction of the gap between levels before the answer flips (it buys noise margin, not immunity).
- [quantm-learning] Photonic matmul uses a weight-stationary broadcast-and-weight crossbar (activations broadcast across rows, a tunable weight at each intersection, columns sum downward); since light power can't go negative, signed ternary weights need a two-rail/two-pipe differential scheme where the weight's sign fixes which pipe the light takes, set once at load time.
- [llm-training] EC2 `user_data` only runs at an instance's first boot, and Terraform treats it as an in-place-updatable attribute by default — so editing the boot script silently updates the attribute without re-running it; `user_data_replace_on_change = true` (or a one-off `-replace`) is required to actually get a fresh boot.
- [llm-training] tmux on the remote box decouples a process's lifetime from the SSH connection — closing the laptop lid drops SSH (which does not auto-reconnect) but tmux + `claude` keep running server-side; a foreground `claude` with no tmux gets SIGHUP-killed on disconnect.
- [llm-training] Stopping an EC2 instance halts the compute charge but the EBS root volume keeps billing regardless of instance state, and an Elastic IP is only free while attached to a *running* instance — a stopped instance's EIP starts costing.
- [llm-training] A GitHub deploy key meant to be read-only was actually registered write-enabled (the "Allow write access" box was left checked at setup) — the safety property the design relied on never held until audited; verify a key's real scope on GitHub rather than trusting the setup doc.
